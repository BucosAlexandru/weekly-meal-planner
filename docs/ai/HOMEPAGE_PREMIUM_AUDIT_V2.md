# Homepage Premium Audit — v2

_Audited: 2026-05-28_
_Scope: the deployed v2 homepage (`/` after `app.js` executes), i.e. what users see in the latest Vercel deploy_
_Owner: Claude Code (Lead Developer)_

---

## What "v2" means here

The static markup in `public/index.html` is largely a fallback. Within milliseconds of `DOMContentLoaded`, `applyTranslations()` (`public/js/app.js:2731`) calls:

1. `renderPremiumHero()` → replaces the static hero with the v2 hero (title + stats row + phone mockup).
2. `renderProductPreview()` → injects "How it works" (3 steps).
3. `renderDiscovery()` → injects "Explore and get inspired" (2 cards).
4. `renderPlannerAnchor()` → injects "Start planning — free".
5. `renderPricingSection()` → re-renders the Free vs Premium cards (hidden for premium users).

The audit below focuses on the **rendered** v2 surfaces, not on the static fallback.

## Findings

### 🔴 High

#### 1. `renderProductPreview()` Step 3 contradicts the pricing claim in 13 locales

Step 3 of the "How it works" v2 section describes the PDF download:

| Lang | Step-3 description (`app.js:1804+`) | Says |
|------|--------------------------------------|------|
| `en` | "...free **once per day**." | Daily cap |
| `es` | "...gratis **una vez al día**." | Daily cap |
| `fr` | "...gratuit **une fois par jour**." | Daily cap |
| `de` | "...einmal **täglich kostenlos**." | Daily cap |
| `pt` | "...gratuito **uma vez por dia**." | Daily cap |
| `ru` | "...бесплатно **один раз в день**." | Daily cap |
| `it` | "...gratuito **una volta al giorno**." | Daily cap |
| `tr` | "...**günde bir kez** ücretsiz." | Daily cap |
| `ar` | "...مجانًا **مرة في اليوم**." | Daily cap |
| `zh` | "...**每天免费一次**." | Daily cap |
| `ja` | "...**1日1回**無料." | Daily cap |
| `ko` | "...**하루 한 번** 무료." | Daily cap |
| `hi` | "...**दिन में एक बार** मुफ्त." | Daily cap |
| `ro` | "PDF elegant, gata de printat..." | (no claim) |

**Reality (per `app.js:484, 500-504, 1735-1747` and i18n `pdf.free.label`):** There is no daily download cap. Free users get the same 2-day preview every time they generate, unlimited times. The Pricing card in the same render cycle already correctly states "✅ Free preview — 2 of 7 days" (see `renderPricingSection()` `app.js:2832`).

**Impact:** The same v2 page contradicts itself between "How it works" and "Free vs Premium". The "PRICING_CONSISTENCY_FIX" (commit `105831bc7`) caught the daily-cap myth on pricing cards + FAQ but missed it inside `renderProductPreview()`.

**Recommended fix:** Replace each language's Step-3 trailing clause with the locale's existing `pdf.free.label` string (e.g. EN → "...beautifully formatted PDF — free 2-day preview, full 7-day plan with Premium."). Or strip the claim entirely (RO already does this without losing readability).

**Files:** `public/js/app.js:1833, 1845, 1857, 1869, 1881, 1893, 1905, 1917, 1929, 1941, 1953, 1965, 1977` → rebuild `app.min.js` → no `npm run content` needed (homepage isn't generated by `generate-content.mjs`).

#### 2. Hero v2 has no Premium upgrade path

`renderPremiumHero()` puts a free CTA (`#hero-cta-btn` → scrolls to planner) and a ghost link (`Explore menus →`). The stats row reads `175+ Recipes · 14 Languages · 0€ Forever free` (per locale).

Effects:
- The €3/month tier is invisible above the fold and through the entire visual sell flow.
- The "0€ Mereu gratuit / Free Forever / 0€ Toujours gratuit" stat actively communicates "there is no paid tier" — at odds with Premium existing.
- The first Premium signal a user encounters is the pricing card section, which sits below the empty planner table — many scroll-screens down.

**Recommended fix:**
- Swap the third stat from "0€ Forever free" to something that lets Premium coexist (e.g. EN: `"€3/mo · Premium"` paired with a small `<a href="#pricing-section">` jump-link). Or drop the third stat and use the freed space for a discreet "From €3/month" line under the CTA.
- Optionally add a secondary CTA in `hero-premium-cta` (currently has primary `Create My Free Plan` + ghost `Explore menus →`) — a third "See Premium →" ghost link.

**Files:** `public/js/app.js:2282-2603` (`copy` object — stat3n / stat3l per locale), `app.js:2622-2641` (markup).

### 🟡 Medium

#### 3. Hero v2 phone mockup overstates what the free tier delivers

The mockup (`app.js:2645-2669`) renders a "PDF ↓" button next to the "Weekly Plan" title in the phone screen, plus 4 rows of meals. The visual promise is: "we generate a PDF of your weekly plan, tap the button". The free PDF is a 2-day preview, not the full week shown.

**Recommended fix:** Either (a) tag the PDF button with a tiny "Premium" pill in the mockup, or (b) show 2 visible meal rows + 2 visibly faded/locked rows to mirror the actual free preview.

**Files:** `app.js:2645-2667`.

#### 4. Premium state does not persist across reloads

`window.hasUnlimited = false` on every page load (`app.js:3031`). Premium users must re-enter their email at `#access-card` on every visit before pricing section hides and free badges disappear. The v2 upsell flow runs against paying customers every refresh.

**Recommended fix:** After a successful `/api/check-access`, store the email in `localStorage` (`mp:lastEmail`). On `DOMContentLoaded`, silently re-call `/api/check-access` with that email; if `active`, flip `hasUnlimited` and re-run `applyTranslations()` so the pricing section hides without user action. Treat the localStorage value as a hint, not an authorization — the server stays authoritative.

**Files:** `public/js/app.js:3023-3115` (verify handler), `app.js:2798-2809` (Stripe success branch — already a good place to seed the localStorage hint after first purchase).

#### 5. "Cuisine Discover" block is static RO-only inside the v2 flow

`<!-- HP_CUISINE_DISCOVER:START -->` (`index.html:146-220`) sits between the hero and the v2 product-preview section. It contains hard-coded RO copy: `Bucătării`, `Franța`, `Japonia`, `Mexic`, `Grecia`, `Italia`, `India`, `Quiche Lorraine · Ratatouille · Supă de ceapă franțuzească`, `Vezi toate cele 46 bucătării`. No data-i18n keys, so it never translates.

Not directly a Premium element, but it sits inside the v2 sell flow — a Korean user on `/` sees Romanian copy between the v2 hero and the localized product-preview/discovery sections. This undercuts the "14 languages" promise that the v2 hero just made one scroll-screen above.

**Recommended fix:** Replace the static markup with another `app.js` render function (e.g. `renderCuisineDiscover()`) called from `applyTranslations()`, OR translate the country names / dish lists / heading into the 14 locales (likely overkill — go with the render function).

**Files:** `public/index.html:146-220`, new render function in `public/js/app.js`.

#### 6. `#manage-subscription` button is wired twice

`portal.js:27` attaches a delegated global click handler for `#manage-subscription`. `app.js:3082` then attaches an additional inline `onclick` after a successful verify. Both fire on the same click → two `POST /api/create-portal-session` requests → two Stripe portal sessions created (one orphaned). The first one's `window.location.href` assignment wins the redirect race.

**Recommended fix:** Delete the inline `manageBtn.onclick = ...` in `app.js:3082-3099` and rely on `portal.js`; or vice-versa. (Inline path passes `return_url`; portal.js doesn't. If you keep portal.js, add `return_url` there.)

**Files:** `public/js/app.js:3082-3099`, `public/js/portal.js:27-39`.

### 🟢 Low

#### 7. Hero badge "Free · No account · 14 languages" is not Premium-accurate

Premium activation is email-based — effectively an account, even if there's no password. "Fără cont" / "No account" technically describes the free tier only. Minor honesty caveat; consider "No signup" or "Use without signing up".

**Files:** `public/js/app.js:2283, 2306, 2329, …` (the `badge` field per locale, lines ~2282-2603).

#### 8. Dead code in v2 sources

`renderLandingFeatures()` (`app.js:1761`) and `renderHowItWorks()` (`app.js:2692`) are defined but never called from `applyTranslations()`. Either wire them or delete — they currently ship in the minified bundle as dead weight (`app.min.js` is ~bundle size constrained — see CLAUDE.md note about the ~1.7 MB JS budget that motivates the `recipes-budget.js` external split).

**Files:** `public/js/app.js:1761-1802`, `2692-2729`.

#### 9. Pricing section position is buried in v2

Scroll order on a fresh visit: hero v2 → cuisine discover → product preview → discovery → planner anchor → empty planner table → empty shopping list (collapsed) → **pricing cards**. That's ~6 scroll-screens before the first detailed Premium pitch. The "Premium" nav link is the only mitigation but is visually small.

Not a bug, but a conversion-funnel observation that pairs with finding #2.

#### 10. The free badge on `#generate-btn` competes with the v2 hero's "Free Forever" promise

`updateButtonState()` (`app.js:1731-1751`) adds `(2/7 days)` to the Generate PDF button label for non-premium users. Good and accurate. But combined with the hero stat `"0€ · Forever free"`, the user-side mental model becomes "the whole thing is free, this button just shows the preview length, no need to pay". Conversion suffers.

Recommend addressing alongside finding #2 (drop the misleading hero stat).

## Suggested fix order

1. **Finding #1** (product-preview daily-cap myth, 13 locales) — pure copy change, no logic risk, fixes a self-contradiction inside one page.
2. **Finding #6** (double-wired manage-subscription) — small code change, removes orphan Stripe portal sessions.
3. **Finding #4** (premium state doesn't survive reloads) — small UX change, big retention win for paying users.
4. **Finding #2** (hero v2 has no Premium signal) + **Finding #10** (`0€ Forever` stat) — small copy + markup change; biggest conversion lift.
5. **Finding #3** (phone mockup PDF promise) — visual tweak alongside #2.
6. **Finding #5** (Cuisine Discover RO-only) — bigger work; can be deferred until a translation pass is scheduled.
7. **Findings #7, #8, #9** — cleanup, defer.

No payment / API / sitemap changes required for any of the above. All fixes are scoped to `public/js/app.js` (rebuilt to `app.min.js`) and `public/index.html`.
