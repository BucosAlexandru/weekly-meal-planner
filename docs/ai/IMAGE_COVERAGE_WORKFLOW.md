# Image Coverage Workflow

This document is the contract between the human-driven part of image curation
(picking photos that look right) and the automated part (optimising,
dropping in the right path, verifying SSR).

Source of truth for what's missing: `docs/ai/IMAGE_QA_REPORT.md`. Re-generate
with `node scripts/audit-images.mjs` after every batch.

## Why a workflow exists at all

The sandbox `claude.ai/code` runs in cannot reach any food-photo host:
`upload.wikimedia.org`, `img.spoonacular.com`, `images.unsplash.com`,
`images.pexels.com`, `cdn.pixabay.com`, `themealdb.com`, `cloudinary.com`,
`images.weserv.nl` — every one returns HTTP 403 `host_not_allowed`. Only
`github.com` / `raw.githubusercontent.com` respond.

That means Claude cannot fetch new images. The human has to bring the bytes
into the sandbox — by committing files, by hosting them on a reachable URL,
or by running the existing external image-fetch tool (which writes
`public/js/recipe-images.js` separately).

The workflow below makes the "after the bytes arrive" half mechanical and
auditable.

## Two scripts

### 1. `scripts/add-recipe-image.mjs` — one recipe at a time

```bash
# from a local file
node scripts/add-recipe-image.mjs --id 12 --source ./sources/dhal.jpg

# from a URL the sandbox can reach (github raw, your own CDN if allowed)
node scripts/add-recipe-image.mjs --id 39 --source https://raw.githubusercontent.com/you/food/main/poutine.jpg

# preview only
node scripts/add-recipe-image.mjs --id 149 --source ./pozole.jpg --dry-run

# overwrite an existing public/images/<slug>.{webp,jpg}
node scripts/add-recipe-image.mjs --id 23 --source ./bibimbap.jpg --force
```

What it does:

1. Resolves `--id` → recipe → slug (mirrors `slug()` in
   `scripts/generate-content.mjs` exactly).
2. Reads bytes from `--source` (local file OR URL — fails clearly on 403).
3. Sniffs magic bytes to confirm it's a real JPEG/PNG/WebP, not an HTML
   error page masquerading as one.
4. Optimises via `sharp`:
   - rotates per EXIF
   - resizes to max 1200px wide (no enlargement)
   - converts to WebP @ quality 80
   - typically lands at 60–180 KB
5. Writes to `public/images/<slug>.webp`.
6. Runs `npm run content` so SSR HTML picks up the new file.
7. Asserts the SSR HTML for `/en/recipes/<slug>/` now references the new URL.
8. Re-runs `node scripts/audit-images.mjs`.

Pass `--no-build` if you're calling this from another script that batches
multiple files before rebuilding.

### 2. `scripts/batch-image-add.mjs` — many at a time

```bash
node scripts/batch-image-add.mjs data/image-batch.json
```

`data/image-batch.example.json` shows the format. The batch script:

- Calls `add-recipe-image.mjs --no-build` per entry.
- Rebuilds once at the end (cheaper than 6 rebuilds).
- Prints a coverage delta: `local: 3 → 9   (Δ +6)`.
- Re-runs the audit for the full report.

## Priority order (use `IMAGE_QA_REPORT.md`)

Always work top-down through the priority list:

1. **P0** — flagship recipe + SSR emoji. As of the last audit:
   - id 12 — Dhal (India) 🚩
   - id 39 — Poutine (Canada) 🚩
   - id 149 — Pozole (Mexico) 🚩
2. **Flagship P1** — flagship + SSR `<img>` but quality issue
   (e.g. ramen images 2–4 MB).
3. **Long-tail P1** — every non-flagship SSR-emoji recipe.

## Per-recipe quality bar

Before approving, eyeball-verify each image against this list:

- Subject matches the dish name (not a generic plate, not a different cuisine).
- Cultural fit (e.g. Mexican pozole shows hominy + radish + lime garnish, not
  just a generic stew).
- Composition: dish fills ≥60% of the frame; no watermarks, no faces.
- Crop: works at 4:3 and 1:1 — the focal point should sit near the centre
  so the mobile crop (≈4:3 on iPhone 14) doesn't decapitate the dish.
- Light: bright enough that the dish reads at 312×231 (the smallest size
  Spoonacular/Wikipedia thumbnails go to).
- Licensing: own work, public domain, or CC0 / CC-BY. Note credit in the
  manifest `note` field.

## Verification per recipe

After the batch finishes, for each new image confirm:

1. **Mobile crop** — load `/en/recipes/<slug>/` on an iPhone-SE-class viewport.
   Hero photo should look composed, not stretched, not too dark.
2. **SSR output** — `grep '<img src.*<slug>'` in the generated HTML should
   match the new file.
3. **No fallback emoji** — the SSR HTML still has the emoji as a backup
   layer underneath, but visually the `<img>` covers it on success. Confirm
   the audit moves the row from bucket 3 or 4 → bucket 1.
4. **OG / share preview** — `<meta property="og:image">` should reference
   the new URL. Generator already passes `recipeImgUrl` to the HEAD()
   helper, so this is automatic once the local file exists.

## Anti-regression rules

- Never edit `public/js/recipe-images.js` by hand (it's auto-generated by
  the external tool — `// DO NOT edit manually`).
- Never overwrite an existing local file without `--force` plus a real
  reason in the commit message.
- Never commit `.env`, source images (use `./sources/` and `.gitignore`),
  or raw downloads bigger than what `add-recipe-image.mjs` produces.
- After every batch, the audit's "mapped" + "wikipedia" + "spoonacular"
  counts in `IMAGE_QA_REPORT.md` must not decrease. We only ADD coverage.

## Commit shape

Small batches (1–6 recipes each), one commit per batch:

```
feat(images): add curated hero photos for Dhal, Poutine, Pozole (P0 flagship)

Coverage: local 3 → 6 (Δ +3). SSR emoji: 108 → 105.
Sources: <credit lines, one per image>.
```

Keep the commit body short; the audit report tells the story.
